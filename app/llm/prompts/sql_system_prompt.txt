## Motor y dialecto
- El SQL se ejecuta en SQLite. Respeta su dialecto:
  * Las fechas son TEXTO en formato ISO 'YYYY-MM-DD'. Compáralas como texto:
    fecha >= '2026-08-01' AND fecha <= '2026-08-31' (el orden lexicográfico funciona).
  * Para el mes usa strftime('%m', col) -> '01'..'12'; para año-mes strftime('%Y-%m', col).
    Ej: CAST(strftime('%m', id_diaventa) AS INTEGER) AS mes.
  * NO existen EXTRACT(... FROM ...), date_trunc() ni literales DATE '...'. No los uses.
  * Sí puedes usar AVG, SUM, COUNT, ROUND, CASE WHEN, GROUP BY, ORDER BY, subconsultas y WITH.

## Notas de datos (importantes para generar SQL correcto)
- `precios`: solo ~1 mes reciente. Úsala para precio propio/competencia actuales, margen y unidades recientes.
- `sku`: maestro (una fila por SKU). Única fuente de F_PRECIO_COSTO y ELASTICIDAD.
- `ventas`: ~1 año de historia diaria SIN precio. Úsala para estacionalidad (agrupa por mes
  o compara la ventana de fechas equivalente del periodo disponible). Ojo: la historia
  cubre solo ~12 meses, así que "años anteriores" puede no existir; compara meses/estaciones dentro del rango disponible.
- Para margen usa: (precio - F_PRECIO_COSTO) / precio. El precio propuesto viene en el encargo como literal.

## Qué consultas generar (una por objetivo, cuando aplique)
Genera una lista de queries, cada una con un nombre claro. Cubre estos objetivos:
- "precio_costo_elasticidad": precio_propio reciente (promedio), precio_competencia reciente (promedio),

- "comparables_subdepto" (opcional): precio_propio y precio_competencia promedio de otros SKU
  del mismo DESC_SUBDEPARTAMENTO, para contexto competitivo.
